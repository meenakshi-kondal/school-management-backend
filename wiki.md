# School Management Backend Wiki

## Overview
This wiki provides a detailed explanation of the backend codebase for the School Management System. The backend is built using Node.js, Express, TypeScript, and MongoDB (with Mongoose). It heavily utilizes a layered architecture pattern dividing responsibilities into schemas, validations, services, controllers, middlewares, and routes to maintain clean separation of concerns and robust maintainability.

## Directory Structure
The application logic lies primarily under the `src/` directory.

- **`config/`**: Contains core configuration scripts, chiefly the database connection setup (`db.ts`).
- **`controllers/`**: Extracts data from HTTP requests, calls appropriate service functions to process it, and crafts the formatted HTTP responses. Includes `authController.ts`, `userController.ts`, `classController.ts`, etc.
- **`helper/`**: Contains helper files or seeding modules such as `adminSeeder.ts`, which ensures an initial administrative user is injected into the database on the server's first load.
- **`interfaces/`**: TypeScript interfaces describing the entity shapes.
- **`middlewear/`**: Custom Express middlewares. Includes `authMiddlewear.ts`, tasked with extracting and verifying JSON Web Tokens (JWT) and performing Role-Based Access Control (RBAC) to protect secure routes.
- **`routes/`**: Handles API endpoint mapping. A single `allRoutes.ts` ties together modular routers like `authRoute.ts`, `userRoute.ts`, `noticeRoute.ts`, and `classRoute.ts`.
- **`schema/`**: Defines the Mongoose Data Models. It manages definitions for `userSchema.ts`, `roleSchema.ts`, `studentSchema.ts`, `teacherSchema.ts`, `classSchema.ts`, `noticeSchema.ts`, etc.
- **`services/`**: Holds all complex business logic, reducing the overall size of the controllers. Direct abstractions of database operations run through these service models (e.g., `authService.ts`, `userService.ts`).
- **`utils/`**: Generic, reusable utility assets such as constants, formatting functions (`commonFunction.ts`), HTTP response formatting wrappers (`response.ts`), and configuration variables (`enum.ts`).
- **`validations/`**: Handlers leveraging `joi` to forcefully validate the structural integrity of incoming request bodies to catch errors before payload data reaches the controllers.

## Core Application Workflow

### 1. Server Entry Point (`server.ts`)
The `server.ts` acts as the bootstrap loader:
- Starts the Express application instance.
- Configures security measures (CORS) and standard JSON body parsers.
- Connects to the database by invoking `connectDB()`.
- Automatically calls the `seedAdmin()` seeder on a successful DB bootstrap.
- Hooks all unified routes into to the core endpoint (`/`).
- Binds to the designated `PORT`.

### 2. Authentication & Authorization
The application uses state-less authentication leveraging JSON Web Tokens.
- **Authentication**: Using endpoints under `/auth`, `authController` triggers `authService` to validate user passwords leveraging `bcryptjs`. On success, tokens are generated by `jsonwebtoken` securely mapping credentials over to the client.
- **Securing Resources**: Protected endpoints invoke the handler in `authMiddlewear.ts`. It verifies the bearer token from the header, appending the validated user’s information inside the request context before it is pushed to the target controller. Appropriate role gates (admin/teacher) verify permissions contextually.

### 3. Database Normalization Pattern
To effectively handle dynamic user variations (Students, Teachers, Admins, Guardians), data is decoupled using normalized schemas:
- **`userSchema`**: Maintains central account credentials (emails, passwords).
- **`roleSchema` / `userRoleSchema`**: Defines custom RBAC system relationships establishing whether an account is a `teacher` or `admin`.
- **Target Collections (`studentSchema`, `teacherSchema`, `guardianSchema`)**: House detailed domain data respective to the entity, establishing a direct relationship referencing the primary `user_id`.

### 4. Code Extension Example
To implement a new feature (e.g., "Library Management"):
1. **Schema**: Create `librarySchema.ts` mimicking Mongoose structure.
2. **Interface**: Create type definitions for TypeScript compilation.
3. **Validations**: Formulate Joi object rules restricting bad inputs into `libraryValidation.ts`.
4. **Service**: Model the business and DB logic internally in `libraryService.ts`.
5. **Controller**: Craft request/response HTTP methods in `libraryController.ts`.
6. **Route**: Define endpoints for `Library` in `libraryRoute.ts`, pass it into `allRoutes.ts`, assigning `authMiddlewear` access validations as logically needed.

## Schema Modeling & Relationships
The backend models data using relational reference concepts inside MongoDB. Rather than embedding documents infinitely, the application stores specialized context in isolated collections (`users`, `students`, `teachers`, `documents`, `guardians`, `classes`) linked via primary object IDs (`ObjectId`). Here is a deeper look into the main models:

### UserModel (`userSchema.ts`)
- **Collection Name:** `users`
- **Purpose**: The central authentication node. Every human interacting with the system is first treated as a generic user.
- **Fields**: Contains exclusively the `email`, encrypted `password`, and an `is_deleted` flag for soft deletions.
- **Relationships**: Instead of the `UserModel` knowing which student/teacher it belongs to, role data (Student, Teacher, Admin) acts identically to a one-to-one or one-to-many relationship where other schemas contain a strict lookup key (`userId`) pointing back to the `users` collection.

### StudentModel (`studentSchema.ts`)
- **Collection Name:** `students`
- **Purpose**: Contains the extensive biological and educational profile information for a student context, such as `name`, `date_of_birth`, `gender`, `roll_number`, `admission_on`, etc.
- **Reference Relationships**:
  - **`userId` (Ref: `users`)**: Links the student back to their login account. This is `required` and `unique`.
  - **`guardian_id` (Ref: `guardians`)**: An ObjectId linking to the `guardians` collection (`guardianSchema.ts`), defining relationships like father/mother/guardian occupation and contact details. This keeps guardian data reusable.
  - **`class_id` (Ref: `classes`)**: Identifies the student's current enrollment class context stored in the `classes` collection.
  - **`photo` & `documents` (Ref: `documents`)**: `photo` securely points to a single `ObjectId`, while `documents` maps to an array `[ObjectId]` in the `documents` collection. This externalizes file assets (storing types, names, remote URLs) effectively decoupling structural constraints.

### TeacherModel (`teacherSchema.ts`)
- **Collection Name:** `teachers`
- **Purpose**: Describes the teaching staff matrix, capturing specific features such as `joining_date`, working `experience`, `name`, `gender`, and basic contact facts `phone`. 
- **Reference Relationships**:
  - **`userId` (Ref: `users`)**: Identical back-pointer logic as `StudentModel`; bridges a teacher's account to their login record.
  - **`class` (Ref: `classes`)**: Unlike students mapping to a primary class, teachers reference an *array* of `[ObjectId]` mapping securely to multiple classes inside the `classes` collection. This enables dynamic assignment of teaching faculties directly to multiple classes.
